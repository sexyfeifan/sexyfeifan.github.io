# HugoDesk ✨📝🚀

把 `Hugo + GitHub Pages` 这套本来有点硬核的流程，做成一个真的能日常用的 macOS 桌面 App。

> 如果你也经历过：写完文章还要切终端、跑 `hugo`、查配置、看 Actions、盯 Pages、处理推送冲突……那 HugoDesk 基本就是为这件事做的。🧃

---

## 📌 这软件到底是干什么的？

一句话：

**HugoDesk = 一个专门给 Hugo 博客写作、配置、检查、发布、排障做的桌面工具。**

它不是“只会写 Markdown 的编辑器”，也不是“只会帮你 push 的外壳”。
它更像是把一整条博客发布链路打包成一个可以点、可以看、可以排查的工作台。💻

你可以在这里完成：

- ✍️ 写 Markdown 正文
- 🖼️ 导入图片并自动放进 Hugo 正确目录
- 👀 在新编辑器里直接预览内容
- 🧠 用 AI 做辅助写作
- 🎨 管理主题与主题设置
- 🔐 配 GitHub Token、远程仓库与发布策略
- 🧪 做发布前检查
- 🚀 一键发布到 GitHub Pages
- 📜 看 Workflow、Pages、日志和故障信息

## 🌟 现在这版有什么亮点？

### 1. 新编辑器真的能用了 ✨

这版最大的变化不是“换了个外观”，而是正文编辑器终于进入可落地状态：

- ✅ 默认使用新的 `Vditor` 编辑器
- ✅ 工具栏恢复完整，不再只留常用项
- ✅ 自带预览能力，旧预览入口已经移除
- ✅ 支持脚注、折叠块等 Hugo/Markdown 写作常用结构
- ✅ 保留 `兼容模式`，需要时可以切回原生编辑器

如果你之前觉得传统文本框写 Hugo 文章不够顺手，这版会明显顺很多。✍️

### 2. 图片导入终于打通了 🖼️

这个功能我知道很关键，所以现在是完整链路：

- 点击编辑器工具栏 `upload`
- 或点击正文编辑区顶部“上传并插入图片”
- 弹出 macOS 文件选择框
- 自动复制到 `static/images/uploads`
- 自动插入 Markdown 图片语法

并且我还处理了一个实际使用里很烦的问题：

**选完图片后插入位置漂移。**

现在会尽量记住选区和光标，再把图片插回原位置。这个细节不做，工具栏看起来能用，实际用起来还是别扭。🔧

### 3. AI 不再只会“排版”了 🧠

现在有两个方向：

- `AI 写作`：独立模块，不属于文本工具
- `AI 检查并排版选区`：保留给兼容模式下的局部修正文稿使用

新的 `AI 写作` 支持：

- 粘贴一段文字
- 粘贴一个链接
- 或者文字 + 链接混合素材

然后它会：

- 识别其中的链接
- 尝试读取可访问网页内容
- 把素材和链接上下文交给你预设的大模型
- 返回一段可直接追加到正文里的 Markdown

这不是简单改错别字，而是把“搜集素材 → 二次整理 → 写进正文”这个过程，往前推了一步。📚

### 4. 发布页依旧是 HugoDesk 的核心价值 🚀

你可以继续在发布页完成：

- 发布前检查
- Pages 来源检查
- Workflow 状态查看
- GitHub 连通性检测
- Git 冲突状态判断
- 日志查看与排障

HugoDesk 的重点一直不是只做“编辑器”，而是做**写完之后真的能发出去、出了问题也知道卡在哪里**。这点不会变。🛠️

## 🧩 这个项目解决了什么问题？

很多人喜欢 Hugo，但现实是：

它很强，也很容易让日常写作流程碎成很多段。

典型流程通常长这样：

1. 打开本地项目
2. 找文章目录
3. 写 Markdown
4. 管 Front Matter
5. 处理图片路径
6. 本地预览
7. 检查配置是否对
8. 查 GitHub Token 是否还能用
9. 推送代码
10. 看 Actions 有没有跑
11. 看 Pages 有没有正常部署
12. 出问题再去翻日志

如果你熟命令行，这些能做。
但问题不是“能不能做”，而是：

**这套流程太重复，也太容易因为一个小环节出错。**

HugoDesk 做的事情，就是把这条链路可视化、结构化，并尽可能减少“做了但心里没底”的时刻。✅

## 🛠️ 核心功能清单

- 📝 文章管理：新建、编辑、保存、删除 Markdown 文章
- 🧾 Front Matter 编辑：标题、摘要、时间、标签、分类、关键词、草稿等
- 🎛️ 双编辑器模式：默认 `Vditor`，保留 `兼容模式`
- 👀 正文预览：由新编辑器自身能力承担
- 🖼️ 图片导入：自动进入 Hugo 静态目录并插入 Markdown
- 🧠 AI 写作：读取素材与链接，生成可追加正文的 Markdown
- 🧹 AI 排版：检查 Markdown 结构、修正符号、清理噪音字符
- 🎨 主题检测与切换：扫描主题目录并按能力显示设置项
- 🔐 凭证管理：Classic + Fine-grained 双 Token 支持
- 🧪 发布前检查：配置、部署、主题、文章数量、冲突、连通性等
- 🌐 GitHub Pages 检查与修复
- 📊 Workflow 状态追踪：最新运行信息自动同步
- 📜 发布日志与错误排查

## 🧠 这个软件的开发思路

我对 HugoDesk 的设计方向一直比较明确，不是堆功能，而是围绕下面几件事做：

### 1. 先保证链路完整，再谈体验

很多“博客工具”最大的问题是：
只覆盖了一小段流程。

比如只做编辑，不管发布。
或者只会 push，不管 Pages。
或者能检查，但定位不到问题。

HugoDesk 的目标是：

**从写作到发布到排障，尽量都在一个上下文里完成。**

### 2. 优先做“确定性”而不是“炫技”

Hugo 和 GitHub Pages 的问题，很多都不是大 bug，而是：

- 配置不一致
- Token 权限不对
- Workflow 路径不对
- Pages 来源不对
- 本地目录结构不标准
- 推送和远端状态不一致

所以这个项目很多地方看起来像“检查器”或者“流程约束器”，这不是保守，而是因为它真的能减少出错。🔍

### 3. 写作体验要升级，但不能脱离 Hugo 现实

新编辑器这次为什么要继续做？
因为原来的原生文本框虽然稳，但写作手感上限确实不高。

但我没有把方向做成“富文本博客工具”，而是依然围绕：

- Markdown 可控
- Hugo 可落地
- Git 仓库可维护
- 图片路径、主题结构、发布逻辑都不失真

所以你现在看到的是：

**更好写，但不脱离 Hugo 本身。**

### 4. AI 要解决真实工作，不只是“看起来高级”

AI 在 HugoDesk 里不是为了放一个按钮，而是解决两种真实场景：

- 文稿结构乱，Markdown 写坏了，需要修
- 手头有素材和链接，想快速生成一段可用正文

如果 AI 只是聊天，那对这个软件没价值。
如果 AI 能直接缩短“整理素材”和“修正文稿”的时间，那它就有价值。⚙️

## 🗂️ 页面结构

- `项目`：项目目录、远程地址、凭据、Hugo 工具、配置包
- `写作`：文章列表、元数据、AI 写作、正文编辑
- `主题设置`：主题检测、切换、参数编辑
- `AI 设置`：模型地址、模型名、API Key
- `发布`：预检、执行、日志、Pages/Workflow 状态

---

## 📦 目录约定

- `latest/`：当前最新可用版本产物
- `HugoDeskArchive/versions/<version>/`：历史版本归档

通常会包含：

- `HugoDesk.app`
- `HugoDesk-vX.Y.Z.dmg`
- `HugoDesk-vX.Y.Z-source.zip`
